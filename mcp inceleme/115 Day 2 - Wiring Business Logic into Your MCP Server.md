**⭐ 🔹 FastMCP Server Boilerplate & Initialization**
→ Writing an MCP server essentially involves writing boilerplate code to wrap your existing business logic using libraries provided by Anthropic. In this lab, it is done inside a new Python module called `accounts_server.py`. The process begins by importing `FastMCP` and the business logic, then initializing the server object.
```python
from mcp.server.fastmcp import FastMCP
from accounts import Account

# Create a new FastMCP server instance with a specific name
mcp = FastMCP("accounts_server")
```
> ⭐ **EXAM NOTE:** Understanding how to initialize a `FastMCP` server instance is a fundamental step in creating custom Python MCP servers. You should know that `FastMCP` is the primary, high-level class used from Anthropic's Python SDK to quickly stand up a server.

**⭐ 🔹 Defining MCP Tools (`@mcp.tool()`)**
→ To expose standard Python functions to an LLM via MCP, you decorate them with the `@mcp.tool()` decorator. The LLM relies entirely on the standard Python docstrings and type hints to understand what the tool does, what arguments it requires, and when to use it. The tool functions themselves usually do not contain heavy logic; they simply delegate the work to the underlying business logic (`accounts.py`).
```python
@mcp.tool()
def get_balance(name: str) -> float:
    """Get the cash balance of the given account name."""
    # Delegates to the business logic
    return Account.get(name).balance

@mcp.tool()
def buy_shares(name: str, symbol: str, quantity: int, rationale: str) -> float:
    """Buy shares of a stock."""
    # Logic to fetch account and execute the buy_shares method...
```
> ⭐ **EXAM NOTE:** The `@mcp.tool()` decorator is a critical concept. You must know that MCP relies heavily on standard Python docstrings and type hints to automatically generate the tool schemas that are sent to the LLM. If your docstrings are poor, the LLM will not know how to use your tools.

**⭐ 🔹 Defining MCP Resources (`@mcp.resource()`)**
→ Resources are another core MCP primitive. While less commonly used than tools, they are valuable for exposing readable data to an LLM. They are defined using the `@mcp.resource()` decorator, which binds a function to a specific, pseudo-URI/URL string pattern. When the client requests that specific URI, the server executes the function and returns the string data.
```python
@mcp.resource("accounts://{name}/strategy")
def read_strategy_resource(name: str) -> str:
    # Extracts the {name} variable from the URI
    account = Account.get(name.lower())
    return account.strategy
```
> ⭐ **EXAM NOTE:** Distinguishing between Tools and Resources is a highly testable MCP concept. **Tools** are for executing actions or dynamic computations that take complex arguments, while **Resources** are used to provide the LLM with static or semi-static read-only context, accessed via a URI-like string pattern.

**⭐ 🔹 Running the MCP Server (`mcp.run()`)**
→ The final required piece of an MCP server script is the execution block at the bottom of the file. When the script is executed by the client, it must start listening for requests. This is done by calling the `.run()` method on the FastMCP instance and specifying the transport mechanism. 
```python
if __name__ == "__main__":
    # Launch the server using standard input/output for communication
    mcp.run(transport='stdio')
```
> ⭐ **EXAM NOTE:** You must know that `stdio` (Standard Input/Output) is the default transport mechanism used to connect local MCP servers to MCP clients. The client runs the server as a background subprocess and communicates back and forth with it using standard text streams.
