**🔹 Environment Variable Configuration (`.env`) for Agent Behavior**
→ Before running the trading floor simulation, several configuration settings must be defined in the `.env` file to control how and when the agents operate. The instructor highlights three specific variables:
1. **`RUN_EVERY_N_MINUTES`**: Dictates the frequency at which the agent traders run. Setting this to `60` means the agents will execute their trading logic every 60 minutes. If omitted from the file, it defaults to 60.
2. **`RUN_EVEN_WHEN_MARKET_IS_CLOSED`**: A boolean flag that determines if agents should continue running outside of standard market hours (e.g., overnight or on weekends). The instructor sets this to `true` specifically to allow the simulation to run and execute trades on a Sunday. 
3. **`USE_MANY_MODELS`**: A boolean toggle controlling the underlying LLMs powering the agents. 
   - If `False`, the system defaults to using a single model (`gpt-4o-mini`) for all agents.
   - If `True`, the system utilizes multiple different models via the DeepSeek API and OpenRouter, allowing agents to run on varied intelligence engines like DeepSeek, Gemini, and Grok.

**🔹 Agent Instantiation (`create_traders`)**

**⭐ 🔹 OpenAI SDK Trace Registration (`add_trace_processor`)**
→ To observe what the agents are thinking and doing, the script must intercept the internal logs generated by the OpenAI Agents SDK. This is done by registering a custom trace processor at the very beginning of the main execution function. The instructor uses a custom class called `LogTracer` (which was built to capture traces and save them to a database) and passes it into the SDK's trace registration method. 

```python
# Registers the custom tracer with the OpenAI Agents SDK
# to intercept all agent thought processes and tool calls
add_trace_processor(LogTracer())
```
> ⭐ **EXAM NOTE:** Understanding how to register a trace processor using `add_trace_processor` is critical for observability in the OpenAI Agents SDK. It is the standard mechanism used to extract, log, and visualize the internal steps, tool calls, and logic of an agentic loop.

**⭐ 🔹 Concurrent Multi-Agent Execution Loop (`asyncio.gather`)**
→ The core engine of the trading floor is an asynchronous continuous loop (`run_every_n_minutes`). Instead of running each agent one by one (synchronously), the system uses Python's asynchronous I/O capabilities to run all four agents concurrently. This is achieved using `asyncio.gather()`, which fires off the `.run()` method for every trader in the list simultaneously. 
Because agents spend a significant amount of time waiting for I/O operations (like API responses from LLMs or data returns from MCP servers), this multi-threaded approach ensures that when one agent is locked/waiting on I/O, another agent is free to execute its logic.

```python
while True:
    # Checks if market is open or if the override flag is set
    if RUN_EVEN_WHEN_MARKET_IS_CLOSED or is_market_open():
        # Kicks off all agent runs simultaneously in parallel
        await asyncio.gather(*[trader.run() for trader in traders])
    
    # Pauses the loop based on the configured minute interval
    await asyncio.sleep(RUN_EVERY_N_MINUTES * 60)
```
> ⭐ **EXAM NOTE:** The use of asynchronous execution (`asyncio.gather`) is a foundational architectural pattern for building efficient, parallel multi-agent systems. You must understand that async allows agents to run concurrently without blocking the entire system while waiting for network/I/O responses from models or tools.

**🔹 Observing the Agentic Trading Floor (UI & Behavior)**
→ Once the script is running, the UI immediately begins populating with real-time data streaming from the agents via the intercepted traces stored in the database. The UI provides several key observability features:
- **Individual Agent Panels:** Each agent (e.g., Warren using GPT-4.1 Mini, George using DeepSeek V3, Ray using Gemini 2.5 Flash) has its own dedicated dashboard showing its unique activity.
- **Trace Visualization:** The internal "thoughts" and actions of the agents fly by in a scrolling log window.
- **Color-Coded Tool Execution:** The logs use different colors to represent calls to different MCP servers. For example, text appearing in **red** specifically indicates that an agent is calling the homegrown "accounts business logic" MCP server to interact with portfolio data.
- **Audio Notifications:** The UI is programmed to play a notification noise whenever one of the agents makes a definitive trading decision (executing a buy or sell).
- **Portfolio State:** The bottom of the UI tracks real-time state changes, including current stock holdings, quantities, portfolio values, and a history of recent transactions (e.g., "Sold 5 AMD shares", "Bought 4 Nvidia shares").
